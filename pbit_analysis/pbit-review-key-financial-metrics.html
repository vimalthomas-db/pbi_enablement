Key Financial Metrics Dashboard — PBIT review

We unpacked Key Financial Metrics Dashboard.pbit and wrote this up for the Databricks / Genie thread. One heads-up: the report’s Metric Table in Power BI is not the same thing as Databricks Metric Views (that’s a UC feature). More on that below.

TL;DR

The dataset already hits Databricks: four facts in gold_custom, with host / warehouse / catalog passed in as parameters. That part’s in decent shape.

Where it gets heavy is the Power BI Metric Table: rows are stuffed in through M (compressed JSON), and 31 DAX measures hang off it, mostly SWITCH logic on metric keys. Labels come from Definitions, same pattern. There are six report pages, 64 visuals, and no RLS. If we want Genie and BI to agree, the bridge rows and the date math need to live in UC + SQL first, then we can put Databricks Metric Views on top for the named metrics people actually query.

Same word, two meanings

Worth spelling out once

What we call itWhat it actually is
Power BI Metric Table A hand-rolled table in this model. Data arrives via ugly M (JSON blob). All the interesting DAX sits here. Has nothing to do with Databricks licensing or features.
Databricks Metric Views The real product: metrics defined in Unity Catalog that SQL and Genie can see. That’s the endgame for “one definition everywhere.”
semantic.* (working names) What we’d probably create in UC to replace the blob: tables like semantic.metric_catalog / semantic.metric_definition. Rename to whatever fits your standards.
Definitions in PBI Tiny dimension hooked to Budget for friendly metric names. Fine in the short term; long term it belongs in Delta like everything else.

What’s working

  • Data You’re already on gold_custom through the Databricks connector with parameters. No fight there.
  • Design Stuffing trend logic into one Metric Table with SWITCH is messy, but at least it’s one messy place. Easier to inventory than logic scattered across 40 measures on random tables.
  • Labels Definitions + Budget give you human-readable metric names. That’s a decent seed list for a real metric_definition table later.
  • Lag The M steps that trim facts to “today minus two days” are documenting a real operational choice. Copy that idea into SQL so warehouse and report don’t drift.

Pain points

  • DAX The same date gymnastics (often anchored on Discharge[Discharge_Date]) are copy-pasted across measures. Nobody wants to reverse-engineer that for Genie without a written spec.
  • JSON in M Bridge and definition data aren’t in the lake yet. Stewards can’t query them; lineage tools barely see them.
  • Dates Lots of auto-generated local date tables. Works until it doesn’t. A single warehouse calendar would be easier to reason about.
  • Cruft Test3 hard-codes a date. Delete it or someone will trust it in prod.
  • RLS There isn’t any. Fine if the workspace is locked down; not fine if you open Genie to a wider group without thinking through UC row filters.
  • Refresh banner Last Refresh is basically “when my laptop ran the query.” If finance cares about SLA, wire the banner to job or table metadata instead.

If you only do four things

Get the Metric Table rows and the time logic into UC + SQL. Then add Databricks Metric Views so Genie isn’t learning from DAX. Flip to How to migrate for the sequence.
  • Land bridge + definition rows in Delta (pick schema names your team won’t hate).
  • Write the “what counts as current month / PY / MTD” rules once in views or dbt, not thirty times in DAX.
  • Collapse toward one date dimension in the warehouse when you can do it without breaking visuals.
  • Kill Test3 or quarantine it outside the published dataset.

Migration (this one report)

You asked to push everything material into Databricks: tables in UC, SQL in the middle, Metric Views for the named metrics. Fair. The trick is that the Power BI Metric Table is just a pattern someone invented in the model; renaming a table in Databricks won’t replace it. You need real rows in Delta and real SQL behind them, then Metric Views pointing at that.

Rough target shape

Rough “this becomes that”

Current (Power BI)Target (Databricks)
Power BI Metric Table (blob + DAX) UC tables for the rows (we’ve been calling them semantic.metric_*) plus SQL for filters and grain. Metric Views consume that; you don’t recreate another “metric table” inside the MV product just for the name.
Definitions A small Delta table (merge into catalog if you prefer one table).
SWITCH trend measures CASE in SQL, or split metrics by family if that reads cleaner in Metric Views.
All the VAR PreMaxDate / MTD / PY stuff Views or dbt macros with one agreed as_of or reporting snapshot so you’re not tuning twelve copies.
Six different auto date tables One calendar in the warehouse (name it what you want; dim_date is boring and fine).

Suggested order of operations

  1. Inventory the Metric Table. Export or trace which columns DAX actually touches. For each SWITCH branch, write down fact table, filter, and grain in plain English (future you will thank present you).
  2. Create the UC tables and load what’s in the JSON today. After that, delete the Binary.Decompress M from the dataset and point Power Query at Delta instead.
  3. Move the calendar math to SQL first. The −2 day cutoffs, month-close quirks, PY/MTD: one place. Metric Views are easier once the underlying views aren’t lying.
  4. Build “report-shaped” views for trends and variances at a stable grain. Reconcile one month and one entity set against the old PBIX before you declare victory.
  5. Publish Metric Views on those objects. Write descriptions someone non-technical could read; if Atlan is in play, mirror the language there.
  6. Rewire the report (or AI-BI) to the warehouse. Peel DAX off until it’s mostly formatting. Lose Test3.
  7. Sanity-check Genie on the same questions analysts ask today. If answers diverge, it’s usually grain or a filter you forgot to port.

Definition of done

  • No more hiding metric rows inside compressed JSON in Power BI.
  • The old Metric Table pattern is fully replaced by tables + SQL you could query from a notebook.
  • Metric Views are what Genie and BI both hit for “official” numbers.
  • There’s a short written note somewhere (Confluence, repo README, whatever) that says how lag and month boundaries work. Doesn’t have to be fancy.
